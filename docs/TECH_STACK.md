# CBIR Engine — Technology Stack Comparison
**Framework, database, vector search, caching, embedding model, and containerization choices — July 2026**

Each section compares options across Pros / Cons / Performance / Scalability / Community / Ease of Deployment / Cost / Recommendation for this specific project: a multi-tenant CBIR SaaS doing image ingestion, embedding, ANN search, reranking, and serving via API.

---

## 1. FastAPI vs Django

| | **FastAPI** | **Django (+ DRF)** |
|---|---|---|
| **Pros** | Async-native (ASGI) from the ground up; built for exactly the "call an embedding model, call a vector DB, call an LLM reranker" I/O-bound workload this product needs; Pydantic V2 (Rust-backed) gives fast, type-safe validation; automatic OpenAPI/Swagger docs; minimal, composable, easy to keep lean for a pure API product | Batteries-included: ORM, migrations, admin UI, auth, permissions all ship out of the box; extremely mature, huge third-party package ecosystem; DRF is a solid, well-understood pattern for CRUD-heavy APIs; excellent for the parts of the product that *are* classic CRUD (tenant management, billing, catalog admin) |
| **Cons** | Lighter on built-in tooling — auth, admin, ORM choice all left to you; younger ecosystem, fewer battle-tested third-party packages than Django; no built-in admin panel (real cost for internal tooling/support teams) | WSGI-first heritage — async support has improved through Django 5.x but ORM calls still often run in a threadpool under the hood rather than truly async; a single request holding a worker for the 2–8 seconds a reranking/LLM call can take is a real anti-pattern under load |
| **Performance** | Benchmarks commonly show 3–4x higher throughput and much lower latency for I/O-bound endpoints (~25ms avg vs ~100ms avg in typical REST comparisons); a single FastAPI worker doing I/O-bound work can handle what 8–10 Django sync workers handle — directly relevant since every CBIR query fans out to an embedding model, a vector DB, and often a reranker call | Comparable for CPU-bound, DB-bound work where the database — not the framework — is the bottleneck; noticeably weaker once requests involve multi-second waits on external calls (embedding APIs, MLLM rerankers), which is the CBIR query path |
| **Scalability** | Scales horizontally well as stateless async services; natural fit for splitting into microservices (ingestion service, query service, rerank service) that can each scale independently | Scales fine for the monolith itself, but scaling out a Django app to isolate the expensive AI-calling parts of the request path is more awkward than in an async-native framework |
| **Community** | Large and fast-growing; strong for AI/ML integration patterns specifically (widely used for ML model serving in 2026); smaller long-tail of tutorials/SO answers than Django | Largest, oldest Python web community; most Stack Overflow answers, most third-party packages, most enterprise track record |
| **Ease of deployment** | Simple to containerize as a lean ASGI service (Uvicorn/Gunicorn+Uvicorn workers); no batteries to configure you don't need | Slightly heavier to deploy correctly (settings, static files, migrations, admin) but well-documented and predictable |
| **Cost** | Lower infrastructure cost at the same load, because fewer/smaller instances handle the same concurrent I/O-bound traffic — a real line-item saving at scale | Higher compute cost per unit of I/O-bound throughput; admin/ORM productivity can reduce *engineering* cost early on, which partially offsets this |
| **Recommendation** | **Use FastAPI as the core query/serving API** — the CBIR request path (embed → ANN search → rerank) is exactly the I/O-bound, concurrency-heavy workload FastAPI is built for. | **Use Django only if/when you need a full admin backoffice** (tenant management, catalog moderation UI, billing) — and consider it as a *secondary* service alongside FastAPI rather than the core API framework. A hybrid (FastAPI for the CBIR API, a small Django or admin-only service for internal tooling) is a legitimate and common 2026 pattern; don't build the whole product in Django. |

## 2. PostgreSQL vs MongoDB

Framed here as PostgreSQL (with `pgvector`, and JSONB for flexible fields) vs. MongoDB (with Atlas Vector Search) — since both now offer native vector capability, this is as much a "one database or two data models" decision as a pure vector-search decision.

| | **PostgreSQL (+ pgvector, JSONB)** | **MongoDB (+ Atlas Vector Search)** |
|---|---|---|
| **Pros** | Single system for relational metadata (tenants, users, billing, catalog structure), flexible JSONB fields for per-tenant custom attributes, *and* vector search — no dual-write problem, no cross-database consistency window; mature transactional guarantees; `pgvector` HNSW performance has improved substantially (0.8+) | Native flexible document model is a strong fit if per-tenant catalogs have wildly different attribute schemas (a fashion catalog and an industrial-parts catalog look nothing alike); Atlas Vector Search integrates vector queries into the same aggregation pipeline as metadata filters and full-text search; higher raw write throughput |
| **Cons** | Rigid schema by default (mitigated by JSONB, but still less natural than MongoDB for wildly heterogeneous per-tenant metadata); vector search extension, not a purpose-built vector engine — will eventually hit throughput/latency limits vector-native DBs don't have | Weaker at complex relational joins and multi-table transactional integrity (billing, tenant permissions); analytical/aggregate queries (reporting, admin dashboards) are markedly slower than PostgreSQL at scale; SSPL licensing for self-managed MongoDB is a consideration if you ever need to self-host rather than use Atlas |
| **Performance** | Strong analytical query performance (10,000–18,000 TPS for GROUP BY-style queries over 100M rows in recent benchmarks); ~95% ANN recall with millisecond HNSW search reported up to 10M vectors; storage is notably more compact (~55% less disk than MongoDB for equivalent record counts) | Higher raw insert/write throughput (20,000–35,000 simple inserts/sec vs Postgres's 15,000–25,000 TPS); weaker on GROUP BY/analytical workloads (roughly 3,000–6,000 ops/sec vs Postgres's much higher analytical throughput) |
| **Scalability** | Vertical scaling is excellent; horizontal scaling (sharding) is less native than MongoDB's, though extensions and managed offerings (Citus, Supabase, RDS read replicas) close most of the gap for this project's likely scale (sub-100M vectors for most tenants) | Native horizontal sharding and multi-region global clusters out of the box — a genuine advantage if you expect to distribute data geographically or need to scale writes aggressively across regions |
| **Community** | Extremely large, extremely mature; the default choice across most of the modern AI-app stack in 2026 specifically because of `pgvector`'s rapid improvement | Very large, mature, strong momentum in AI-app circles due to Atlas Vector Search and "autoEmbed" — but ecosystem is more oriented to document/analytics use cases than transactional catalog + billing data |
| **Ease of deployment** | Trivial via managed options (RDS, Supabase, Neon, Cloud SQL); one system to operate instead of two/three | Easy via Atlas managed service; self-managed MongoDB is more operationally involved and comes with the SSPL licensing consideration |
| **Cost** | Managed Postgres (Supabase Pro, RDS) is meaningfully cheaper at comparable tiers (~$25/mo vs MongoDB Atlas M10 at ~$57/mo for equivalent starting tier); consolidating metadata + vectors into one system also removes a whole separate bill/operational surface | Atlas pricing is higher at equivalent tiers; the "single unified platform" pitch is real but you're paying a premium for it |
| **Recommendation** | **Use PostgreSQL as the system of record** for tenants, catalogs, billing, and item metadata, with JSONB for per-tenant custom attributes. Use `pgvector` for small-to-mid tenant vector workloads (below ~10–50M vectors per tenant) to avoid a second system entirely. | **Reach for MongoDB only if a specific tenant's catalog is so heterogeneous that JSONB genuinely becomes painful**, or if you need MongoDB's native geo-sharding for a specific customer. For most CBIR SaaS customers, this is not the default choice — Postgres wins on the "one system, lower cost, sufficient vector performance" tradeoff. Large-scale vector search (100M+ per tenant) should go to a dedicated vector DB (see next section), not to either of these general-purpose databases. |

## 3. Qdrant vs Milvus vs FAISS

These aren't mutually exclusive at every layer — FAISS is a *library*, Qdrant and Milvus are full *databases* (some of which use FAISS-like algorithms internally). Comparing them as the three realistic choices for the ANN recall layer.

| | **Qdrant** | **Milvus** | **FAISS** |
|---|---|---|---|
| **Pros** | Rust-based, memory-efficient; best-in-class metadata filtering and *filtered* vector search (critical for "similar items in stock, under $50, in category X"); simple to self-host; generous free tier | Purpose-built for massive scale; disaggregated storage/compute/coordination architecture scales independently; multiple index types (HNSW, IVF, IVF-PQ) for tunable recall/speed/memory tradeoffs; strongest choice once you're past ~500M vectors | Not a database — a battle-tested, GPU-accelerable similarity search *library* from Meta; maximum control, zero network overhead if embedded directly in your service; no licensing or hosting cost at all |
| **Cons** | Below purpose-built billion-scale systems on raw distributed scale; smaller community than Milvus or Pinecone | Heaviest operational footprint of the three — requires etcd, object storage (S3/MinIO), and message queues in distributed mode; real DevOps investment to run well | No persistence, replication, access control, or filtering built in — you must build all of that yourself; no multi-tenancy, no metadata store; a "search algorithm," not a "search service" |
| **Performance** | 15–30ms in-memory query latency typical at moderate scale; excels specifically at filtered search scenarios where competitors see 2–3x slowdowns | Leads on raw write throughput due to distributed architecture; competitive query latency (often <10ms p50 in benchmarks) at very large scale; latency spread vs. Qdrant/Pinecone is single-digit milliseconds below ~50M vectors | Fastest possible in-process search when embedded directly (no network hop); GPU-accelerated variants push this further; but "performance" here excludes everything a database has to do around the search itself |
| **Scalability** | Handles up to hundreds of millions of vectors comfortably; teams needing true billion-vector, heavy-replication workloads (e.g., Reddit Engineering's public evaluation) have found Milvus scales more gracefully at the extreme end | Explicitly built for billion-scale; the clear pick once any single tenant or shared index needs to go past the several-hundred-million-vector range | Scales only as far as the machine(s) you embed it in — no native distributed clustering; scaling is a DIY exercise (sharding, replication all on you) |
| **Community** | Fast-growing, strong Rust/infra-engineering following; 70+ integrations across embedding providers, orchestration, and observability tools | Very large (44K+ GitHub stars), broad enterprise adoption (10,000+ teams cited), strong managed-cloud option (Zilliz) | Enormous, long-standing (Meta FAIR), the default academic/research baseline; huge amount of prior art and benchmarking literature |
| **Ease of deployment** | Straightforward single-binary self-host or managed Qdrant Cloud; low operational overhead relative to Milvus | Most complex of the three to self-host (multi-component distributed system); Zilliz Cloud (managed Milvus) meaningfully reduces this burden | Trivial to embed in a Python/C++ service; zero "deployment" in the database sense, but that means you own everything a database would otherwise provide |
| **Cost** | Best value at small-to-mid scale: free tier (1GB forever) plus low-cost paid tiers (~$25/mo); self-hosted infra cost is modest (~$100–200/mo range for 10M vectors) | Self-hosting is the most engineering-hours-expensive of the three; Zilliz Cloud pricing is competitive for the scale it targets but you're paying for genuine billion-scale capability you may not need yet | Zero licensing cost; the "cost" is entirely engineering time to build persistence, filtering, and ops tooling around it |
| **Recommendation** | **Start here.** For a new CBIR SaaS operating per-tenant catalogs that will mostly sit well under 100M vectors, Qdrant gives the best combination of filtered search (essential for catalog/e-commerce use cases), reasonable cost, and low operational burden. | **Plan a migration path to Milvus/Zilliz Cloud for scale-out.** Once any single tenant or shared index approaches the hundreds-of-millions range, or once true billion-vector, high-replication requirements appear, migrate that workload to Milvus — its disaggregated architecture is specifically built for this and most vector DBs (including Zilliz) offer tooling to migrate from Qdrant. | **Use FAISS only as an internal library**, e.g., inside offline batch re-ranking, evaluation/benchmarking pipelines, or a lightweight local dev/test environment — not as the production-facing vector store, since you'd otherwise be rebuilding persistence, filtering, and multi-tenancy that Qdrant/Milvus already provide. |

## 4. Redis vs No Cache

| | **Redis (caching layer)** | **No cache** |
|---|---|---|
| **Pros** | Sub-millisecond reads for hot data (popular queries, frequently-requested embeddings, session/auth tokens, rate-limit counters); dramatically reduces repeated load on the vector DB and embedding service for popular/repeated queries (duplicate images, trending catalog items); doubles as a job/task queue backend (e.g., for async embedding pipelines) and pub/sub for real-time indexing status | Radically simpler architecture — one fewer moving part, one fewer thing to operate, monitor, and keep consistent; no cache-invalidation bugs (a very real class of production incident: "why is this stale result still showing after we re-indexed?") |
| **Cons** | Added operational component: another service to run, monitor, secure, and scale; introduces cache-invalidation complexity — especially relevant here because re-indexing/re-embedding a catalog item must reliably bust any cached results referencing it; another potential single point of failure if not run with replication | Every repeated query re-hits the embedding model and/or vector DB and/or reranker at full cost and full latency — meaningful waste for a CBIR product where duplicate/similar queries (popular products, trending images) are common |
| **Performance** | Turns a ~20–100ms ANN + rerank round-trip into a <1ms cache hit for repeated queries; also ideal for caching precomputed query embeddings (skip re-running the encoder for an identical query image) and hot rate-limit/auth checks | Consistent but strictly worse tail latency and throughput ceiling under any repeated-query load pattern; every request pays the full embedding + ANN + rerank cost regardless of how common the query is |
| **Scalability** | Scales horizontally (Redis Cluster) and takes real load off the more expensive downstream systems (GPU embedding inference, rerank model calls) — this is often the difference between needing 1 GPU inference replica and needing 4 | Downstream systems (embedding inference, vector DB, reranker) must be sized to handle 100% of traffic at full cost, with no cheap absorption layer for spikes or popular-query traffic |
| **Community** | Extremely mature, ubiquitous, huge ecosystem, well understood failure modes and tooling (Redis Enterprise, managed offerings from every cloud) | N/A |
| **Ease of deployment** | Very easy to add (managed Redis on every major cloud, or a single container); the complexity is in *cache design/invalidation strategy*, not in standing up Redis itself | Trivially "easy" by virtue of not existing — but this is a false economy once query volume grows |
| **Cost** | Low direct cost (a small managed Redis instance is cheap relative to GPU embedding inference costs) and a strong *net cost reducer* once it's offloading expensive embedding/inference/rerank calls | Zero direct cost, but higher effective cost per query at scale because every request pays full embedding + ANN + rerank compute, including GPU inference time — this is usually the more expensive path once volume is non-trivial |
| **Recommendation** | **Use Redis.** For this product specifically — where query embedding and MLLM-based reranking are the expensive steps — caching (a) precomputed query embeddings for repeated/identical query images, (b) full result sets for popular queries, and (c) rate-limiting/auth state is a high-value, low-risk addition. Treat cache invalidation on re-index as a first-class design requirement from day one, not an afterthought. |

## 5. OpenCLIP vs DINOv2 vs SigLIP (2)

| | **OpenCLIP** | **DINOv2** | **SigLIP 2** |
|---|---|---|---|
| **Pros** | Open-source, reproducible CLIP training/weights; huge range of pretrained checkpoints across scales and datasets; native text-image joint embedding space enables text-to-image and image-to-text queries out of the box; broad candidate retrieval (good recall across diverse categories, higher mAP@k in several benchmarks) | Self-supervised — no labels/captions needed at all; captures visual *structure* (composition, pose, spatial layout, texture) exceptionally well; strong top-1 precision on fine-grained, visually-similar categories (species, parts, textures) — 5x better than CLIP-family on some fine-grained species benchmarks | Best semantic understanding among the three; improved localization and dense features vs. original SigLIP/CLIP; multilingual out of the box (NaFlex support for arbitrary aspect ratios); strong cross-modal (text↔image) alignment for compositional/text queries |
| **Cons** | Weaker on fine-grained/structural similarity than DINOv2 (e.g., far behind DINOv2 on fine-grained species classification in independent benchmarks); image-only similarity quality is sometimes described as less calibrated than dedicated vision-only encoders, though recent research disputes a strong "intra-modal misalignment" penalty | No native text-image alignment — can't do text-to-image or compositional text queries without pairing it with a separate text encoder; weaker on broad-category recall in some scenarios (a geometric issue, not semantic: won't easily surface "different-looking but conceptually related" images) | Newer, so slightly less third-party tooling/community content than CLIP; largest model variants (up to 1B params) cost more to run than compact CLIP/DINOv2 checkpoints at the same throughput target |
| **Performance** | Solid, well-benchmarked general-purpose retrieval performance; result quality is dataset/training-data dependent given the many available OpenCLIP checkpoints | Wins decisively on structural/fine-grained visual similarity tasks (scene classification, texture, pose); best pure image-to-image similarity in several independent benchmarks | 2–3 percentage point improvement over SigLIP on zero-shot classification/retrieval benchmarks; independent studies show SigLIP-family can outperform DINOv2 on image-image similarity classification despite being trained with an inter-modal (not pure image) objective |
| **Scalability** | Available in many sizes (ViT-B to ViT-bigG); easy to pick a checkpoint matching your latency/throughput budget | Available from small to large ViT variants; being self-supervised, adapts well to fine-tuning on unlabeled domain-specific image collections (useful for verticals with proprietary catalogs and no captions) | Available from 86M to 1B parameters; NaFlex support means no forced resizing/cropping distortion, which helps at scale across heterogeneous image sizes |
| **Community** | Very large open-source community (LAION, OpenCLIP maintainers), extensively used across the industry as the default "good enough" baseline | Backed by Meta, strong research adoption, growing production adoption specifically for retrieval/similarity tasks | Backed by Google, newer but rapidly gaining adoption; strong recent independent benchmarking coverage |
| **Ease of deployment** | Widely supported in every major ML serving framework (Hugging Face, TorchServe, Triton); easiest "default choice" to get running quickly | Equally well supported via Hugging Face/Meta releases; slightly more work if you need a paired text encoder for hybrid search | Well supported via Hugging Face and Google's release tooling; NaFlex variable-aspect-ratio handling requires slightly more thought in the preprocessing pipeline |
| **Cost** | Compute cost scales with chosen checkpoint size; mid-size ViT-B/L checkpoints are cheap to run at scale | Similar cost profile to CLIP at equivalent model size; self-supervised pretraining cost is irrelevant to you since you're using released weights | Similar cost profile; larger variants (1B) cost more per image embedded — pick a smaller NaFlex variant unless dense/localization features are specifically needed |
| **Recommendation** | **Use as the default multimodal (text+image) embedding** for any query path that supports typed/text queries or compositional search — it's the safest, most broadly-supported baseline. | **Use as a second, structural embedding** specifically for verticals/use cases where fine-grained visual similarity matters more than semantic category (industrial parts, textures/materials, fashion pattern-matching, near-duplicate/forensic use cases). | **Prefer SigLIP 2 over OpenCLIP as the primary encoder if starting fresh in 2026** — it consistently edges out CLIP-family models on both semantic retrieval and (per some benchmarks) image-image similarity, plus native multilingual support is valuable for a global SaaS customer base. **Recommended overall approach: run SigLIP 2 as primary encoder, and add DINOv2 as a secondary embedding for verticals/query types where structural similarity outperforms semantic similarity** — an ensemble/routing strategy rather than picking one exclusively, consistent with what production systems are converging toward. |

## 6. Docker vs Podman

| | **Docker** | **Podman** |
|---|---|---|
| **Pros** | Best-in-class developer experience (Docker Desktop, Compose); largest ecosystem — virtually every tutorial, CI template, and Stack Overflow answer assumes Docker; BuildKit remains excellent for build caching and multi-stage builds; the default mental model every new hire already knows | Daemonless and rootless *by default* — no privileged background process, meaningfully smaller attack surface (11 default kernel capabilities vs Docker's 14); Apache 2.0 licensed with no commercial-use cost cliff at any team size; native Kubernetes alignment — generates K8s YAML directly (`podman generate kube`), closely mirrors the containerd runtime that ~95% of K8s clusters actually run in production |
| **Cons** | Docker Desktop's paid licensing tier (~$24/seat/month) applies to organizations above 250 employees / $10M revenue — a real, growing line-item cost as the company scales; root-daemon architecture is a materially weaker default security posture unless rootless mode is explicitly configured (and it usually isn't) | Smaller community and ecosystem than Docker (~32K vs ~71K GitHub stars as one rough proxy); some Docker-specific tooling/extensions need adaptation; background/long-running containers need explicit systemd/Quadlet setup rather than "just working" the way a daemon-backed tool does; team ramp-up requires some muscle-memory adjustment even though CLI compatibility is high |
| **Performance** | Mature BuildKit build-cache performance is excellent for complex multi-stage Dockerfiles (relevant here for images bundling ML dependencies/model weights) | Reported 30–35% faster CI build/startup times in some head-to-head tests, attributable to the lack of a daemon; particularly noticeable in high-volume CI pipelines spinning up many short-lived containers |
| **Scalability** | Runs everywhere or containerd underneath in most managed Kubernetes offerings anyway, so Docker at the *build* layer doesn't block scaling the *runtime* layer | Pod-native model and closer alignment with containerd (the actual K8s runtime standard) reduces friction specifically when scaling to a Kubernetes-based production deployment — relevant since a multi-tenant CBIR SaaS will likely run on K8s |
| **Community** | Largest, most mature container ecosystem by a wide margin; ~67% of developers report using Docker per recent surveys | Smaller but rapidly growing and increasingly backed by Red Hat/Fedora/RHEL defaults, SUSE, and Canonical; the default engine on RHEL 9/10 |
| **Ease of deployment** | Docker Desktop is the smoothest local dev experience on macOS/Windows; Compose remains the most widely understood local multi-service orchestration tool | Podman Desktop has closed most of the historical GUI gap and is free with no licensing restrictions; rootless builds simplify CI/CD security configuration (no need for privileged Docker-in-Docker) |
| **Cost** | Free for small teams/individuals; real licensing cost (~$24/seat/month) kicks in for larger organizations — for a growing SaaS company this is a cost that scales with headcount, not with infrastructure usage | No licensing cost at any team size — for a startup building a SaaS product with an eye on later headcount growth, this avoids a future line-item entirely |
| **Recommendation** | **Fine to use for local development** if your team already knows it well and you're a small team currently under the Docker Desktop licensing threshold — don't force a migration purely on principle if it's not yet costing you anything. | **Prefer Podman for CI/CD and production, especially since you'll likely deploy on Kubernetes.** The combination of (a) zero licensing risk as you grow past 250 employees, (b) a materially better default security posture — genuinely relevant for a SaaS product ingesting customer image catalogs — and (c) closer alignment with the containerd runtime your Kubernetes cluster actually uses, makes Podman the stronger default for this project's CI/production path. A common, low-friction pattern: Docker Desktop (or Podman Desktop) for local dev, Podman for CI/CD and production images — both are OCI-compliant so the same Dockerfiles and images work with either. |

## Summary Recommendation Table

| Decision | Recommended choice | Key reason |
|---|---|---|
| Web framework | **FastAPI** (Django only for optional internal admin) | CBIR query path is I/O-bound (embed → ANN → rerank); FastAPI's async model fits directly |
| Primary database | **PostgreSQL + pgvector** | One system for metadata + billing + vectors at sub-100M-vector scale; cheaper, simpler, sufficient performance |
| Vector search | **Qdrant** now, **Milvus/Zilliz** at extreme scale, **FAISS** only as an internal library | Qdrant's filtered search + low ops burden fits most tenants; Milvus is the scale-out path |
| Caching | **Redis** | Offloads expensive embedding/rerank compute for repeated/popular queries; strong net cost reducer |
| Embedding model | **SigLIP 2 primary, DINOv2 secondary (ensemble/routing)** | SigLIP 2 wins on semantics + multilingual + text queries; DINOv2 wins on fine-grained structural similarity |
| Containerization | **Podman for CI/production, Docker acceptable for local dev** | No licensing risk at scale, better default security posture, closer alignment with Kubernetes/containerd |
